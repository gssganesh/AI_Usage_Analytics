2. THE "WHY" - ARCHITECTURAL DECISIONS & TECH STACK REASONING
--------------------------------------------------------------------------------

A. WHY REACT.JS (FRONTEND)?
   - REASON: We chose React for its Virtual DOM and Component-based architecture. 
     In an analytics app, data changes constantly (due to filters). React 
     efficiently re-renders only the changed charts without refreshing the page, 
     creating a seamless "App-like" feel.
   - ALTERNATIVE: Plain HTML/JS would have been too "clunky" to manage the 
     complex state of 8+ different filter variables.

B. WHY NODE.JS & EXPRESS (BACKEND)?
   - REASON: Speed and Scalability. Node.js uses non-blocking I/O, which is 
     perfect for handling multiple database queries simultaneously when a user 
     refreshes the dashboard. It also shares the same language (JavaScript) as 
     the frontend, ensuring 100% logic consistency.

C. WHY MYSQL (DATABASE)?
   - REASON: Relational integrity. Student data is highly structured (ID, Score, 
     Stream). MySQL allows us to perform "JOIN" operations and complex 
     "GROUP BY" queries (like AI usage by Stream) much faster and more 
     reliably than NoSQL databases like MongoDB.
   - DATABASE CHOICE: We used the 'mysql2' library for 'Promise-based' 
     connections, allowing the backend to "wait" for data safely without 
     crashing.

D. WHY GLASSMORPHISM (THE DESIGN)?
   - REASON: To "Wow" the user. Modern enterprise software is often boring 
     and grey. By using backdrop-filter blur and translucent cards, we 
     created a "Premium" aesthetic that feels like a futuristic OS. This 
     increases user engagement and makes the dense data feel lighter and 
     easier to digest.

3. DETAILED EXECUTION LOGIC: HOW EVERYTHING RUNS
--------------------------------------------------------------------------------

STEP 1: INITIALIZATION
   - When the user opens the URL, the React Frontend sends a "Handshake" 
     request to the Backend `GET /api/stats`.
   - The Backend checks the `.env` file for database credentials and 
     establishes a secure tunnel to MySQL.

STEP 2: THE FILTER PIPELINE (DYNAMIC QUERIES)
   - Every time you change a filter (e.g., set State to 'California'), a 
     new "Query Object" is sent to the backend.
   - The backend dynamically builds a SQL string: 
     "SELECT * FROM students WHERE state = 'California' AND ..."
   - This ensures that the PDF exports and charts are ALWAYS "Live" and 
     synced to your exact view.

STEP 3: PREDICTIVE ENGINE (LOGISTIC REGRESSION LOGIC)
   - REASONING: We chose Logistic Regression for the "Risk Warning" tab. 
   - THE MATH: We use a Sigmoid function. It calculates a "Risk Score" by 
     weighting negative factors (Low CGPA, High AI usage hours) against 
     positive factors (High Digital Literacy). 
   - WHY IT WORKS: Unlike a simple "If-Else" statement, the model provides a 
     smooth probability (0-100%). It captures the "At-Risk" students who 
     haven't failed yet but are "trending" toward failure.

STEP 4: TELEGRAM ALERT SYSTEM
   - When you click "Send Telegram Alert," the frontend hits a specialized 
     Express route.
   - The backend uses the `node-telegram-bot-api` to talk to Telegram's 
     Global Servers. Within milliseconds, a secure encrypted message is 
     pushed to the Admin's phone.

4. OPERATIONAL FLOW: HOW TO RUN & EXECUTE
--------------------------------------------------------------------------------

TO START THE BACKEND:
   1. Open a terminal in the `/backend` folder.
   2. Commands: `npm install` (once) -> `node server.js`
   3. Success Marker: "Server running on port 5000" and "Database Connected!"

TO START THE FRONTEND:
   1. Open a terminal in the `/frontend` folder.
   2. Commands: `npm install` (once) -> `npm run dev`
   3. Success Marker: Dashboard appears in browser with "LIVE DATA" badge.

THE DATABASE STRUCTURE:
   - Primary Table: `student_ai_dataset_csv`
   - Key Columns: `cgpa`, `ai_tool`, `academic_stream`, `exam_score`, `assignment_completion_pct`.
